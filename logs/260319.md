# ROS2 Humble + Ignition Fortress 自定义赛道工程规范

> 适用环境：Ubuntu 22.04 / ROS2 Humble / Ignition Gazebo 6 (Fortress) / TurtleBot4

---

## 一、问题复盘与根因分析

### 问题 1：LiDAR 点云在墙壁处连续漂移
**根因**：SDF `<polyline>` 被拆分为独立线段，接缝处法线不连续，LiDAR 光线在接缝处穿透或反射异常。  
**解决**：用 Python 将 SVG path 挤出为 **Collada (.dae) mesh**，单一连续几何体，无接缝。

---

### 问题 2：SVG 解析失败（脚本输出空 world）
**根因**：SVG 使用了 `xmlns` 命名空间前缀，`ElementTree` 的 `findall('polyline')` 无法匹配带命名空间的标签；同时赛道使用的是 `<path>` 元素而非 `<polyline>`。  
**解决**：用 `strip_ns()` 剥离命名空间，支持 `path/polyline/polygon/line` 全部元素类型。

---

### 问题 3：path 解析崩溃（ValueError: could not convert string to float: 'c'）
**根因**：SVG path 中包含贝塞尔曲线命令（`c/C/s/S/q/Q`），原脚本只处理了直线命令。  
**解决**：完整实现所有 SVG path 命令解析，曲线用 `BEZIER_STEPS=12` 细分为折线。

---

### 问题 4：STL 文件无法被 Ignition 加载
**根因**：Ignition Fortress 6 的 `STLLoader` 对 numpy 生成的二进制 STL 法线字节序有已知 bug。  
**解决**：改用 **Collada (.dae)** 格式，XML 文本格式，解析器完全不同，无此问题。

---

### 问题 5：DAE 加载导致 Segfault（normal count 不匹配）
**根因**：DAE 中 `<triangles>` 使用双索引（`offset="0"` + `offset="1"`），dartsim 要求 normal count == vertex count，双索引下法线数为 0 导致空指针访问。  
**解决**：将 `NORMAL` 移入 `<vertices>` 块，改为**单索引**，每个面展开独立顶点保证 1:1 对应。

---

### 问题 6：赛道尺寸过小，机器人在赛道外
**根因**：SVG 坐标系使用 mm 单位，`MM_TO_M=0.001` 直接换算导致赛道只有 0.07m 宽，小于 TurtleBot4（0.351m）。  
**解决**：用内外墙间距自动计算 scale，目标赛道宽 = TurtleBot4 宽 × 4 = 1.404m；同时计算赛道中心偏移，平移到 Gazebo 原点。

---

### 问题 7：Visual already exists + Segfault（模型重复注册）
**根因**：将完整 world 文件（含 ground_plane + walls）覆盖官方 `custom_track_lidar.sdf`，但 `world name` 写的是 `track` 而非 `custom_track_lidar`，launch 构建 topic 路径时拼出 `/world/custom_track_lidar.sdf/...`，`.` 是非法字符导致 parameter_bridge 崩溃；同时 SceneBroadcaster 在 GUI 重连时重复注册已有 visual，触发渲染器空指针。  
**解决**：world name 必须与文件名完全一致；不覆盖整个 world 文件，只追加 wall model。

---

### 问题 8：Sensors 插件冲突导致传感器父节点找不到
**根因**：在 world 文件里添加了 `ignition-gazebo-sensors-system` 插件，而机器人 URDF 内部已经触发了同一插件，两次加载导致渲染树父节点错乱，LiDAR 等传感器无法创建。  
**解决**：world 文件**不加** Sensors 插件，与官方 maze.sdf 保持一致，传感器由机器人 URDF 自行驱动。

---

### 问题 9：双 GPU 环境下 Segfault（libGLdispatch）
**根因**：系统有 Intel 核显 + NVIDIA 独显，Ignition ogre2 渲染器默认走 Mesa 软件渲染，与 NVIDIA 驱动冲突。  
**解决**：启动时指定使用 NVIDIA：
```bash
__NV_PRIME_RENDER_OFFLOAD=1 __GLX_VENDOR_LIBRARY_NAME=nvidia ros2 launch ...
```

# 赛道墙壁连续性问题总结

> 接续上一份工程规范，专项记录"墙壁断口"问题的完整排查过程

---

## 问题现象

Gazebo 中赛道墙壁在某处出现明显断口（空洞），不是阴影，是真实的几何缺口。

---

## 根因分析（三层）

### 第一层：SVG 路径未闭合

**现象**：`parse_path` 解析后首尾点距离 = 2.54mm / 3.84mm

**原因**：Inkscape 导出的 path `d` 属性末尾没有 `z` 命令，起点和终点在画图时没有精确对齐

**验证**：
```bash
grep -o 'z"' track.svg | wc -l   # 应为2，否则没有z
```

**解决**：用 Python 给所有 path 的 d 属性末尾加 `z`：
```python
import re
with open('track.svg') as f: content = f.read()
def add_z(m):
    d = m.group(2).rstrip()
    if not d.endswith(('z','Z')): d += ' Z'
    return m.group(1) + d + m.group(3)
content = re.sub(r'(d=")([^"]+)(")', add_z, content)
with open('track.svg','w') as f: f.write(content)
```

---

### 第二层：`parse_path` 的 `Z` 命令 bug

**现象**：加了 `z` 之后，解析后首尾距离仍然不为 0

**原因**：`z` 被 `is_cmd()` 识别为命令字符后，下一个 token 不是数字，循环再次进入 `is_cmd` 分支更新了 `cmd`，`elif cmd in 'Zz'` 永远不会执行

**验证**：
```python
sp = parse_path("m 0,0 l 10,0 l 0,10 z")[0]
print(len(sp), sp[-1])  # 修复前：3个点尾=(10,10)；修复后：4个点尾=(0,0)
```

**修复**：在 `is_cmd` 分支里检测到 `Z/z` 时立即处理：
```python
if is_cmd(t):
    cmd = t; idx += 1
    if cmd not in 'Zz': last_cp = None
    if cmd in 'Zz':                      # Z 无参数，立即处理
        if cur: cur.append((sx, sy))     # 闭合到 M 命令记录的精确起点
        subpaths.append(cur); cur=[]
        cx,cy=sx,sy; cmd=None; continue
    continue
```

**注意**：闭合点用 `(sx, sy)` 而不是 `cur[0]`——贝塞尔曲线细分后 `cur[0]` 可能有浮点误差，`sx,sy` 是 `M/m` 命令记录的精确起点。

---

### 第三层：挤出函数把路径当线段而非闭合环处理

**现象**：路径解析首尾完全重合（距离=0），但 Gazebo 里墙壁仍有断口

**原因**：
- 挤出循环 `for i in range(n-1)` 是线性的
- 第一段的起点法线和最后一段的终点法线各自独立计算，没有相互参照
- 首尾交接处法线不连续，产生几何缝隙

**修复**：把路径当作闭合环处理，使用取模运算 `(i+1) % m_pts`，所有顶点统一用前后两条边法线的平均值：

```python
m_pts = n - 1  # 去掉末尾重复的闭合点，作为环处理

# 每条边的法线
edge_normals = [seg_normal(pts[i], pts[(i+1)%m_pts]) for i in range(m_pts)]

# 每个顶点的法线 = 前后两条边的平均（miter join，无首尾特例）
vert_normals = []
for i in range(m_pts):
    avg = edge_normals[(i-1)%m_pts] + edge_normals[i]
    L = np.linalg.norm(avg)
    vert_normals.append(avg/L if L > 1e-9 else edge_normals[i])

# 逐段挤出，取模自动处理首尾回绕
for i in range(m_pts):
    j = (i+1) % m_pts   # 最后一段自动回到第一点，无需特例
    ...
```

## 关键代码对比

### 修复前（有断口）

```python
# 线性处理，首尾特殊对待
for i in range(n - 1):
    j = i + 1
    nb = seg_norms[j] if j < n-1 else seg_norms[i]  # 最后一段用自己
    if i == 0:
        n_start = na       # 第一段直接用当前段法线，不考虑与最后一段的连接
    if j == n - 1:
        n_end = na         # 最后一段不考虑与第一段的连接
        continue
```

### 修复后（连续闭合）

```python
# 环形处理，所有顶点统一
m_pts = n - 1
edge_normals = [seg_normal(pts[i], pts[(i+1)%m_pts]) for i in range(m_pts)]
vert_normals = []
for i in range(m_pts):
    avg = edge_normals[(i-1)%m_pts] + edge_normals[i]
    L = np.linalg.norm(avg)
    vert_normals.append(avg/L if L > 1e-9 else edge_normals[i])

for i in range(m_pts):
    j = (i+1) % m_pts   # 自动处理首尾回绕，无需特例
```

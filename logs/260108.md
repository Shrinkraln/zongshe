### 1. Micro-ROS 库编译与环境 (Build System)

*   **问题现象**：`tcpdump` 抓不到 UDP 包，报错 `ninja: missing libmicroros.a`，或者库虽然编好了但没有 UDP 发送功能。
*   **根本原因**：
    1.  **手动创建目录导致跳过下载**：为了修复 `ros2_tracing`，提前建立了 `src` 目录，导致构建脚本误以为所有源码已就绪，生成了空的静态库。
    2.  **配置缓存残留**：在未开启 WiFi 传输模式时编译过一次，导致库记住了“无传输”状态。
*   **解决方案**：
    *   **核弹级重构**：`rm -rf components/micro_ros_espidf_component` (连根拔起)。
    *   **正确修复依赖**：重新克隆组件 -> 运行 `idf.py reconfigure` 让它下载 -> **再** 手动修复 `ros2_tracing` (替换为 GitHub 源 + 添加忽略文件)。
    *   **清理缓存**：`idf.py fullclean` + `idf.py reconfigure`。

### 2. 网络通讯与连接 (Connectivity)

*   **问题现象**：PC 能 Ping 通 ESP32，但 `tcpdump` 抓不到 8888 端口的 micro-ROS 数据包，Monitor 报错 `Error Code: 1`。
*   **根本原因**：
    1.  **Menuconfig 配置未生效**：默认的 `rclc_support_init` 似乎未能正确读取 Kconfig 中的 IP 字符串。
    2.  **Wi-Fi 省电模式**：导致 UDP 丢包严重。
*   **解决方案**：
    *   **代码显式配置**：放弃默认初始化，改用 `rclc_support_init_with_options` 并在 C 代码中显式设置 IP (`rmw_uros_options_set_udp_address`)。
    *   **关闭省电模式**：`esp_wifi_set_ps(WIFI_PS_NONE);`。
    *   **重试机制**：在 `do...while` 循环中加入延时重试。

### 3. 代码逻辑 (C Code)

*   **问题现象**：编译报错类型不匹配，或者运行逻辑死锁。
*   **根本原因**：
    1.  **拼写错误**：`rcl_executor_t` 写成了 `rclc_executor_t`。
    2.  **格式化错误**：`printf` 用 `%d` 打印字符串类型的端口号。
    3.  **调试信息位置**：将打印 IP 的代码放在了死循环之后，导致出错时看不到配置信息。
*   **解决方案**：修正类型定义，修正 `printf` 格式，将调试打印移至循环前。

### 4. 可视化与基础设施 (Docker & Web)

*   **问题现象**：Agent 收到数据，但 `ros2 topic echo` 无输出；Rosbridge 启动后 Web 端无数据。
*   **根本原因**：
    1.  **Docker 网络隔离**：Agent 和 Rosbridge 处于不同的网络命名空间。
    2.  **QoS 不匹配**：ESP32 发送的是 **Best Effort**，Web 端默认订阅的是 **Reliable**（互不兼容）。
*   **解决方案**：
    *   **网络统一**：所有容器（Agent, Rosbridge, Client）统一使用 **`--net=host`**。
    *   **QoS 对齐**：在 `ros2` 命令行或 Web JS 代码中，显式指定 `reliability: "best_effort"`。
    *   **数据填充**：在驱动层加入正弦波模拟数据，解决“数据全是0”的视觉误会。
